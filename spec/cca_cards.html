🎯 Core Concepts & Capability Nucleus

Understanding the Foundation of CCA
Card 1 of 11

📋 What is a Capability?

Definition: A cohesive set of functionality that delivers tangible value.

✅ GOOD Examples:

  • ✓ Product Catalog manages product information
  • ✓ Payment Processing handles transactions
  • ✓ Motor Control regulates speed/position
  • ✓ User Authentication manages identity

❌ AVOID:

  • ✗ Database-Access-Capability
  • ✗ UI-Capability
  • ✗ Team-A-Capability
🎯 GOLDEN RULE: Express purpose in ONE sentence. Focus on WHAT, not HOW. Based on functionality, not layers.

🏗️ The Nucleus (3 Layers)

┌─────────────────────────────┐ │ 🌐 ADAPTATION │ │ External Interfaces │ │ REST, gRPC, UART, SPI │ └─────────────┬───────────────┘ │ Uses ┌─────────────▼───────────────┐ │ ⚙️ REALIZATION │ │ Infrastructure │ │ DB, HW, RTOS, Files │ └─────────────┬───────────────┘ │ Implements ┌─────────────▼───────────────┐ │ 💎 ESSENCE │ │ Pure Domain Logic │ │ NO Infrastructure Deps │ └─────────────────────────────┘

📊 Layer Details

Layer Purpose Dependencies
💎 ESSENCE Define WHAT Only Contracts
⚙️ REALIZATION Implement HOW Essence + Infra
🌐 ADAPTATION External API Realization

💎 Essence Layer

Contains:

  • Pure domain logic
  • Core algorithms
  • Business rules
  • Domain state

Rules:

  • ✅ NO database calls
  • ✅ NO file I/O
  • ✅ NO hardware access
  • ✅ ONLY other Capability Contracts

⚙️ Realization Layer

Contains: Database access, Hardware drivers, File I/O, Message queues, RTOS calls, Cache management

Examples: PostgreSQL queries, GPIO/ADC access, Redis caching, Kafka producers

🌐 Adaptation Layer

Contains: REST controllers, gRPC services, Message queue listeners, UART/SPI handlers, WebSocket endpoints

Examples: @RestController classes, gRPC service implementations, MQTT subscribers, CAN bus handlers

💡 KEY INSIGHT: Essence is pure and reusable. Realization adapts to environment. Adaptation exposes to world.

📜 Capability Contracts

Provisions, Requirements & Protocols
Card 2 of 11

🎁 PROVISIONS

Interfaces I PROVIDE to others

Structure:

new Provision(
    "ServiceName",
    ServiceInterface.class,
    "Description of what it does"
)

Example Interface:

public interface ProductService {
    Product getProductDetails(String id);
    List<Product> searchProducts(String query);
}

// Declaration:
new Provision(
    "ProductLookup",
    ProductService.class,
    "Provides product details and search"
)

🔌 REQUIREMENTS

Interfaces I NEED from others

Structure:

new Requirement(
    "ServiceName",
    ServiceInterface.class,
    false,  // optional?
    "Why I need this"
)

Example:

new Requirement(
    "PricingService",
    PricingService.class,
    false,  // REQUIRED
    "Needed for product prices"
)

📡 PROTOCOLS

HOW to interact with me

new Protocol(
    "REST_HTTP",
    "JSON",
    Map.of("latency", "50ms",
           "security", "OAuth2")
)

📝 Complete Contract Example

CapabilityContract contract = 
    new CapabilityContract(
    
    // 🎁 PROVISIONS
    List.of(
        new Provision(
            "ProductLookup",
            ProductService.class,
            "Product details & search"
        ),
        new Provision(
            "ProductEvents",
            ProductEventPublisher.class,
            "Product change events"
        )
    ),
    
    // 🔌 REQUIREMENTS
    List.of(
        new Requirement(
            "PricingService",
            PricingService.class,
            false,  // Required
            "For product prices"
        ),
        new Requirement(
            "InventoryService",
            InventoryService.class,
            false,  // Required
            "For stock levels"
        )
    ),
    
    // 📡 PROTOCOLS
    List.of(
        new Protocol("REST_HTTP", "JSON",
            Map.of("latency", "50ms")),
        new Protocol("gRPC", "Protobuf",
            Map.of("latency", "20ms")),
        new Protocol("DirectCall", "Binary",
            Map.of("latency", "5ms"))
    )
);

🔄 Interaction Pattern

ProductCatalog (Capability A) 🎁 PROVIDES: ProductService 🔌 REQUIRES: PricingService ▲ │ │ Fulfilled by │ Pricing (Capability B) 🎁 PROVIDES: PricingService
💡 CONTRACT DESIGN PRINCIPLES: ✅ Focus on WHAT (outcome), not HOW (implementation) • ✅ Use Semantic Versioning (MAJOR.MINOR.PATCH) • ✅ Keep stable yet flexible • ✅ Support multiple protocols

🔄 Capability Lifecycle

Initialization, Dependencies & State Management
Card 3 of 11

🎭 CapabilityInstance Interface

public interface CapabilityInstance {
    
    // Phase 1: Setup
    void initialize();
    
    // Phase 2: Begin work
    void start();
    
    // Phase 3: Stop work
    void stop();
    
    // Phase 4: Release resources
    void cleanup();
    
    // Provide contract implementation
    Object getContractImplementation(
        Class<?> contractType
    );
    
    // Receive dependencies
    void injectDependency(
        Class<?> contractType,
        Object implementation
    );
}

📊 Lifecycle States

📦 INSTANTIATED ↓ initialize() 🏁 INITIALIZED ↓ injectDependency() 🔌 DEPENDENCIES INJECTED ↓ start() 🚀 STARTED (Runtime) ↓ stop() 🛑 STOPPED ↓ cleanup() 🧹 CLEANED UP

⚙️ Manager Responsibilities

CapabilityLifecycleManager:

  1. Build dependency graph
  2. Topological sort (correct order)
  3. Initialize capabilities
  4. Inject dependencies
  5. Start capabilities
  6. (Runtime operations)
  7. Stop capabilities (reverse order)
  8. Cleanup

💻 Implementation Example

public class ProductCatalogCapability 
    implements CapabilityInstance {
    
    private final ProductCatalogEssence essence;
    private final DatabasePool database;
    
    // Dependencies (injected later)
    private PricingService pricingService;
    private InventoryService inventoryService;
    
    @Override
    public void initialize() {
        // Setup internal state
        database.initializeSchema();
        essence.loadInitialData();
    }
    
    @Override
    public void start() {
        // Begin active operations
        startBackgroundTasks();
        openConnections();
    }
    
    @Override
    public void stop() {
        // Graceful shutdown
        stopBackgroundTasks();
        closeConnections();
    }
    
    @Override
    public void cleanup() {
        // Release resources
        database.close();
        cache.clear();
    }
    
    @Override
    public Object getContractImplementation(
        Class<?> contractType) {
        if (contractType == 
            ProductService.class) {
            return new ProductServiceImpl(this);
        }
        return null;
    }
    
    @Override
    public void injectDependency(
        Class<?> contractType,
        Object implementation) {
        if (contractType == 
            PricingService.class) {
            this.pricingService = 
                (PricingService) implementation;
        }
        // ... other dependencies
    }
}
💡 KEY INSIGHT: Manager handles orchestration automatically. You just implement the interface. Dependencies are injected in correct order based on Contract declarations.

⚡ Efficiency Gradients

Performance Optimization Where You Need It
Card 4 of 11

🎯 Core Concept

Not all code needs same optimization!

Different parts operate at different levels of performance and abstraction.

📊 The Gradient Spectrum

HIGH EFFICIENCY ←→ LOW EFFICIENCY LOW ABSTRACTION ←→ HIGH ABSTRACTION ┌─────────────────────────────┐ │ 🔥 CRITICAL PATH │ │ • Bare-metal code │ │ • Direct HW registers │ │ • Assembly │ │ • RTOS interrupts │ │ ⏱️ Latency: Microseconds │ │ 📌 Use: Motor control, │ │ sensor interrupts │ └─────────────────────────────┘ ┌─────────────────────────────┐ │ ⚙️ STANDARD PATH │ │ • Standard OS services │ │ • Compiled languages │ │ • Optimized libraries │ │ ⏱️ Latency: Milliseconds │ │ 📌 Use: Business logic, │ │ API calls, DB queries │ └─────────────────────────────┘ ┌─────────────────────────────┐ │ 🌈 NON-CRITICAL PATH │ │ • Frameworks (Spring, etc.) │ │ • GC languages (Java, etc.) │ │ • ORMs, JSON parsing │ │ ⏱️ Latency: Seconds (OK) │ │ 📌 Use: Config, logging, │ │ admin tasks │ └─────────────────────────────┘

🎨 When to Use Each

Gradient Use For
High Real-time loops, interrupts, HFT
Medium Most application code
Low Config, logging, admin

💻 Example: Motor Control

🔥 HIGH EFFICIENCY (Critical):

// Runs every 100 microseconds
private void onMotorInterrupt() {
    // Direct HW register access
    int pos = hardwareRegisters
              .readEncoder();
    
    // Inline calculation
    double speed = stateEstimator
                   .estimateSpeed(pos);
    
    // Optimized PID (no allocations)
    double output = pidController
                    .calculate(speed, target);
    
    // Direct PWM write
    pwmDriver.setDutyCycle(output);
}

⏱️ Latency: ~10 microseconds

🌈 LOW EFFICIENCY (Non-Critical):

// Runs on-demand (rarely)
public void updateConfiguration(
    String configData) {
    
    // JSON parsing (high-level)
    JSONObject config = 
        new JSONParser().parse(configData);
    
    // Network call
    Params params = httpClient
        .get("/api/motor-params");
    
    // Update essence
    essence.setTargetSpeed(
        config.getDouble("targetSpeed")
    );
    
    // Database persistence
    database.save("motor_config", config);
}

⏱️ Latency: ~500 milliseconds (acceptable)

💡 KEY INSIGHT: Both methods can exist in the SAME Capability! Apply the right gradient to each function. Don't over-optimize non-critical paths. Don't under-optimize critical paths.

🔄 Evolution Envelopes

Managing Change with Predictability
Card 5 of 11

📊 Semantic Versioning

MAJOR . MINOR . PATCH 2 . 1 . 3
Component When to Increment Impact
MAJOR Breaking changes Incompatible API
MINOR New features Backward compatible
PATCH Bug fixes Backward compatible

📝 Examples

MAJOR (2.x.x → 3.0.0):

  • Remove method from interface
  • Change method signature
  • Change data format
  • Remove entire provision

MINOR (2.1.x → 2.2.0):

  • Add new method to interface
  • Add new provision
  • Add optional parameter
  • Enhance existing feature

PATCH (2.1.3 → 2.1.4):

  • Fix bug in logic
  • Performance improvement
  • Security patch
  • Documentation fix

🎯 Evolution Envelope

public class EvolutionEnvelope {
    String currentVersion;
    String previousVersion;
    List<DeprecationPolicy> deprecations;
    List<MigrationPath> migrations;
}

⚠️ Deprecation Policy

new DeprecationPolicy(
    "Contract v1.x",           // What
    LocalDate.of(2025, 12, 31), // EOL
    "Performance improvements", // Why
    "Use Contract v2.x"        // Replacement
)

Components:

  • Target: What's being deprecated
  • Warning period: How long supported
  • EOL date: When support ends
  • Reason: Why deprecating
  • Replacement: What to use instead

🔄 Migration Path

new MigrationPath(
    "1.5.0",                    // From
    "2.0.0",                    // To
    "https://docs.../migration", // Docs
    "upgrade-script.sh"         // Tools
)

Includes:

  • Code examples
  • Step-by-step guide
  • Automated scripts
  • Testing recommendations
  • Rollback procedures

📋 Version Strategy

DO:

  • ✅ Plan breaking changes carefully
  • ✅ Provide migration paths
  • ✅ Give adequate warning period
  • ✅ Support N-1 versions
  • ✅ Document all changes

DON'T:

  • ❌ Break compatibility without notice
  • ❌ Skip version numbers
  • ❌ Change MAJOR version frequently
  • ❌ Deprecate without replacement
💡 KEY INSIGHT: Evolution Envelopes make change predictable. Consumers know what to expect. Migrations are smooth. System stability is maintained.

💻 Unit Test (Essence)

@Test
void testProductSearch() {
    // Arrange
    ProductCatalogEssence essence = 
        new ProductCatalogEssence();

        essence.searchProducts("Laptop");

    assertEquals("Laptop",

🎯 Golden Rules

  1. ALWAYS package Essence separately
  2. Create feature-based bundles
  3. Use dependency injection

📦 Bundle Structure

🔧 Embedded Bundle

Contains:

  • Essence (pure logic)
  • Embedded Realization:
    • Direct HW access
    • RTOS calls
    • GPIO/ADC/Timers
  • Minimal Adaptation:
    • UART, SPI, CAN Bus, I2C

Use for: IoT devices, Microcontrollers, Industrial control, Automotive systems

Contains:

  • Essence (same pure logic)
  • Enterprise Realization:
    • PostgreSQL/MySQL
    • Redis cache
    • Kafka/RabbitMQ
    • Elasticsearch
  • Full Adaptation:
    • REST API, gRPC, GraphQL, WebSockets

Use for: Cloud platforms, Web applications, Microservices, SaaS products

✅ Patterns & ❌ Anti-Patterns

Best Practices & Common Mistakes
Card 10 of 11

✅ DO - Best Practices

Design:

  • ✅ Define by functional cohesion
  • ✅ One clear responsibility
  • ✅ Express in one sentence
  • ✅ Deliver tangible value

Essence:

  • ✅ Keep pure (no infrastructure)
  • ✅ High test coverage
  • ✅ Reusable across environments
  • ✅ Only depend on Contracts

Contracts:

  • ✅ Use Semantic Versioning
  • ✅ Document breaking changes
  • ✅ Provide migration paths
  • ✅ Support multiple protocols

Efficiency:

  • ✅ Apply gradients strategically
  • ✅ Optimize critical paths
  • ✅ Use abstractions elsewhere
  • ✅ Measure before optimizing

Testing:

  • ✅ Many unit tests (Essence)
  • ✅ Some integration tests
  • ✅ Contract compliance tests
  • ✅ Few E2E tests

Packaging:

  • ✅ Separate Essence package
  • ✅ Feature-based bundles
  • ✅ Dependency injection
  • ✅ Clear module boundaries

Evolution:

  • ✅ Plan breaking changes
  • ✅ Deprecate gracefully
  • ✅ Support N-1 versions
  • ✅ Document migrations

❌ DON'T - Anti-Patterns

Design:

  • ❌ Create by technical layer
  • ❌ Create by team structure
  • ❌ Too many responsibilities
  • ❌ Unclear value proposition

Essence:

  • ❌ Mix infrastructure code
  • ❌ Direct database calls
  • ❌ File I/O in Essence
  • ❌ Hardware access in Essence

Contracts:

  • ❌ Break without notice
  • ❌ Skip version numbers
  • ❌ Change frequently (MAJOR)
  • ❌ Undocumented changes

Efficiency:

  • ❌ Over-optimize everything
  • ❌ Under-optimize critical paths
  • ❌ Ignore performance data
  • ❌ Premature optimization

Testing:

  • ❌ Only E2E tests
  • ❌ No unit tests
  • ❌ Skip contract tests
  • ❌ Test with real infrastructure

Packaging:

  • ❌ Mix Essence with Realization
  • ❌ Monolithic packages
  • ❌ Tight coupling
  • ❌ Circular dependencies

Evolution:

  • ❌ Breaking changes without warning
  • ❌ No deprecation period
  • ❌ Missing migration docs
  • ❌ Support too many versions
