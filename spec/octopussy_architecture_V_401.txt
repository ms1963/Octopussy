1. VISION & POSITIONING

1.1 What Is Octopussy?

Octopussy is a production-grade, Python-native Agentic AI platform built entirely on Capability-Centric Architecture (CCA) 0.2 by Michael Stal. It provides a complete, pluggable, zero-trust runtime for deploying, orchestrating, and operating autonomous AI agents — from a single Raspberry Pi to a Kubernetes cluster.
Every functional unit of Octopussy is a Capability with a strict three-layer Nucleus (Essence / Realization / Adaptation), a formal Contract (Provisions / Requirements / Protocols), a Capability Instance managed by the OctopussyCapabilityLifecycleManager (OCLM), and an Evolution Envelope. No exceptions.

2. ARCHITECTURAL FOUNDATIONS

2.1 Capability-Centric Architecture (CCA 0.2) Application

Every functional unit of Octopussy is a Capability structured according to CCA 0.2 by Michael Stal. The following CCA elements are applied throughout:
CCA 0.2 Terminology Clarification (Normative)

Provisions — the interfaces a Capability provides to other Capabilities (its public API, the services it offers to the outside world).
Requirements — the interfaces a Capability needs from other Capabilities (its dependencies; these references are injected by the OCLM via inject_dependency()).
Protocols — the interaction rules, ordering constraints, invariants, and messaging/transport standards used by the Capability (e.g., NATS.io pub/sub, REST over HTTPS, gRPC over mTLS, RabbitMQ AMQP, MCP 2025-11-25 Streamable HTTP). Protocols describe both behavioral contracts and wire-level communication standards.

Capability Nucleus
Three mandatory layers, present in every Capability:

Essence — pure domain logic; no infrastructure, no I/O, no external dependencies
Realization — technical mechanisms: database adapters, queue implementations, HTTP clients
Adaptation — external interfaces: REST adapters, gRPC adapters, CLI adapters, MCP adapters

Capability Contract
Formal specification of every Capability's interface:

Provisions — what the Capability provides to others (its public interface)
Requirements — what the Capability needs from others (its dependencies, injected via inject_dependency())
Protocols — interaction rules, ordering constraints, invariants, and messaging/transport standards

Capability Instance
A Capability Instance is a fully initialized, running, configured instantiation of a Capability. It is the unit that is registered in the OctopussyCapabilityRegistry, started by the OctopussyCapabilityLifecycleManager, and referenced by other Capabilities via dependency injection. Each Capability Instance:

Is created exactly once by CapabilityFactory
Is registered under a unique name in the OctopussyCapabilityRegistry
Passes through all six CCA lifecycle phases: instantiate → initialize() → inject_dependency() → start() → [runtime] → stop() → cleanup()
Is never shared across system boundaries; each deployment has its own set of instances
Carries its own Evolution Envelope declaring its version, minimum client version, deprecation policy, and migration guide

inject_dependency() — CCA Lifecycle Phase 3 (Normative)
The OCLM calls inject_dependency() on every Capability after initialize() and before start(). During this phase, the OCLM resolves all interfaces declared as Requirements in the Capability's Contract and injects the concrete references into the Capability instance. This is the exclusive mechanism by which Capabilities obtain references to other Capabilities' ports. No Capability may obtain a reference to another Capability's port by any other means (no service locator, no global registry lookup at runtime).
Evolution Envelope
Every Capability carries an Evolution Envelope:

Semantic versioning (MAJOR.MINOR.PATCH)
Deprecation policy (12 months notice before removal of any Contract method)
Migration path (published before old version is removed)
CAPABILITY_VERSION and MINIMUM_CLIENT_VERSION constants

Capability Registry
The OctopussyCapabilityRegistry is the authoritative catalogue of all Capability Instances in the system. It:

Accepts registrations of Capability Instances with their dependency declarations
Detects circular dependencies at registration time using Kahn's algorithm
Rejects non-Capability registrations
Provides topological_order() for startup sequencing
Is itself a CCA element

Capability Lifecycle Manager
The OctopussyCapabilityLifecycleManager (OCLM) orchestrates all Capability Instances through the mandatory six-phase CCA lifecycle. It:

Computes topologically sorted startup order using Kahn's algorithm
Enforces the six-phase lifecycle on every Capability Instance without exception
During inject_dependency(): resolves all Requirement interfaces and injects them into each Capability
Shuts down in strict reverse topological order
Is itself a CCA element

All 26 Capabilities are registered in the OctopussyCapabilityRegistry and orchestrated by the OCLM. Circular dependencies are detected at registration time and rejected immediately with a CircularDependencyError.
The mandatory six-phase CCA lifecycle applies to every Capability without exception:
instantiate → initialize() → inject_dependency() → start()
    → [runtime] →
stop() → cleanup()

Shutdown proceeds in strict reverse topological order.

4.0.2 OctopussyMessage (Immutable Inter-Agent Message)

# octopussy/core/message.py

from __future__ import annotations

import hashlib
import hmac
import time
from dataclasses import dataclass, field
from typing import Any
from uuid import UUID, uuid4

from octopussy.core.enums import MessagePriority, MessageType

@dataclass(frozen=True, slots=True)
class OctopussyMessage:
    """
    Immutable inter-agent message. All inter-agent communication uses this type.
    HMAC-SHA256 signature is mandatory (bytes, never Optional[bytes]).
    TTL is mandatory; expired messages are rejected.
    """
    message_id:   UUID
    message_type: MessageType
    priority:     MessagePriority
    sender_id:    UUID
    recipient_id: UUID
    payload:      dict
    trace_id:     UUID
    timestamp:    float          # Unix timestamp (UTC)
    ttl_sec:      int            # Time-to-live in seconds
    signature:    bytes          # HMAC-SHA256; mandatory

    @classmethod
    def create(
        cls,
        message_type: MessageType,
        priority:     MessagePriority,
        sender_id:    UUID,
        recipient_id: UUID,
        payload:      dict,
        trace_id:     UUID,
        hmac_secret:  bytes,
        ttl_sec:      int = 300,
    ) -> "OctopussyMessage":
        msg_id    = uuid4()
        timestamp = time.time()
        # Sign over canonical fields
        raw = (
            str(msg_id) + str(message_type) + str(priority)
            + str(sender_id) + str(recipient_id)
            + str(timestamp) + str(ttl_sec)
        ).encode()
        sig = hmac.new(hmac_secret, raw, hashlib.sha256).digest()
        return cls(
            message_id=msg_id,
            message_type=message_type,
            priority=priority,
            sender_id=sender_id,
            recipient_id=recipient_id,
            payload=payload,
            trace_id=trace_id,
            timestamp=timestamp,
            ttl_sec=ttl_sec,
            signature=sig,
        )

    def verify_signature(self, hmac_secret: bytes) -> bool:
        raw = (
            str(self.message_id) + str(self.message_type) + str(self.priority)
            + str(self.sender_id) + str(self.recipient_id)
            + str(self.timestamp) + str(self.ttl_sec)
        ).encode()
        expected = hmac.new(hmac_secret, raw, hashlib.sha256).digest()
        return hmac.compare_digest(self.signature, expected)

    def is_expired(self) -> bool:
        return time.time() > self.timestamp + self.ttl_sec

from __future__ import annotations
from abc import ABC, abstractmethod
from uuid import UUID

from octopussy.core.enums import AgentState

class ObservabilityPort(ABC):
    @abstractmethod
    def start_span(self, name: str, trace_id: UUID, attributes: dict) -> object: ...
    @abstractmethod
    def log(self, level: str, message: str, context: dict) -> None: ...
    @abstractmethod
    def metric(self, name: str, value: float, labels: dict) -> None: ...
    @abstractmethod
    def audit(self, event: str, actor: object, resource: str, outcome: str, details: dict) -> None: ...

class SecurityPort(ABC):
    @abstractmethod
    async def authenticate(self, token: str) -> dict: ...
    @abstractmethod
    async def authorize(self, user_id: UUID, resource: str, action: str) -> bool: ...
    @abstractmethod
    async def scan_prompt(self, prompt: str) -> bool: ...
    @abstractmethod
    async def validate_output(self, output: str, policy: str) -> bool: ...

class TokenBudgetPort(ABC):
    @abstractmethod
    async def check_budget(self, agent_id: UUID, period: str) -> bool: ...
    @abstractmethod
    async def record_usage(self, agent_id: UUID, input_tokens: int, output_tokens: int) -> None: ...

class AgentLifecyclePort(ABC):
    @abstractmethod
    async def transition_state(self, agent_id: UUID, new_state: AgentState) -> None: ...
    @abstractmethod
    async def get_state(self, agent_id: UUID) -> AgentState: ...
    @abstractmethod
    async def record_heartbeat(self, agent_id: UUID) -> None: ...

class ToolExecutorPort(ABC):
    @abstractmethod
    async def execute_tool(self, agent_id: UUID, tool_id: str, args: dict) -> dict: ...

class SandboxPort(ABC):
    @abstractmethod
    async def provision(self, agent_id: UUID, spec: dict) -> str: ...
    @abstractmethod
    async def execute(self, agent_id: UUID, command: list[str], timeout_sec: int = 30) -> dict: ...
    @abstractmethod
    async def destroy(self, agent_id: UUID) -> None: ...

class MessageSignerPort(ABC):
    @abstractmethod
    def sign(self, message: object, secret: bytes) -> bytes: ...

    def verify(self, message: object, signature: bytes, secret: bytes) -> bool: ...

    @abstractmethod
    async def store(self, agent_id: UUID, key: str, value: str) -> None: ...
    @abstractmethod
    async def retrieve(self, agent_id: UUID, key: str) -> str | None: ...
    @abstractmethod
    async def search(self, agent_id: UUID, query: str, top_k: int = 5) -> list[dict]: ...
    @abstractmethod
    async def delete(self, agent_id: UUID, key: str) -> None: ...
    @abstractmethod
    async def clear(self, agent_id: UUID) -> None: ...

class ConfigurationPort(ABC):
    @abstractmethod

    async def rotate_secret(self, name: str) -> str: ...
    @abstractmethod
    async def delete_secret(self, name: str) -> None: ...

from __future__ import annotations
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Any

# ─────────────────────────────────────────────
# Immutable request/response dataclasses
# ─────────────────────────────────────────────

@dataclass(frozen=True, slots=True)
class LLMRequest:
    """Immutable LLM request."""
    messages:     tuple[dict, ...]
    model:        str
    temperature:  float = 0.7
    max_tokens:   int   = 4096
    system:       str   = ""
    trace_id:     str   = ""

@dataclass(frozen=True, slots=True)
class LLMCompletion:
    """Immutable LLM completion."""
    content:       str
    model:         str
    input_tokens:  int
    output_tokens: int
    finish_reason: str

# ─────────────────────────────────────────────
# LLMPort ABC — base for all LLM providers
# ─────────────────────────────────────────────

class LLMPort(ABC):
    """
    Abstract base for all LLM provider ports.
    Delivered by LLMRouterCapability via resolve_llm_port().
    Every concrete implementation below is a Provision of LLMRouterCapability.

    An LLMPort defines either a cloud-provided LLM (e.g., Google Gemini, Anthropic Claude,
    OpenAI GPT) or a local LLM provided by a local inference engine (e.g., Ollama,
    llama.cpp, LM Studio, SGLang, vLLM, MLX, HuggingFace Transformers).
    Octopussy assumes that the inference engine provides a REST API that it calls via httpx.
    It does not need to care about the underlying GPU hardware leveraged by the inference engine.
    API tokens and credentials are always retrieved from SecretsCapability.
    """
    @abstractmethod
    async def complete(self, request: LLMRequest) -> LLMCompletion: ...

class OctopussyAgent(ABC):
    """
    Abstract base for all Octopussy agents.
    Subclasses: StatelessOneShot, StatefulLoop, ScheduledTaskDaemon.
    All agents are created by AgentFactoryCapability, never directly.
    """

    def __init__(
        self,
        spec:           AgentSpec,
        hmac_secret:    bytes,
        lifecycle:      AgentLifecyclePort,
        obs:            ObservabilityPort,
        security:       SecurityPort,
        token_budget:   TokenBudgetPort,
        tool_executor:  ToolExecutorPort,
        llm_port:       LLMPort,
        vlm_port:       VLMPort | None = None,
        rag:            RAGPort | None = None,
        graph:          GraphPort | None = None,
        memory:         MemoryPort | None = None,
    ) -> None:
        self._spec          = spec
        self._agent_id      = spec.agent_id
        self._hmac_secret   = hmac_secret
        self._lifecycle     = lifecycle
        self._obs           = obs
        self._security      = security
        self._token_budget  = token_budget
        self._tool_executor = tool_executor
        self._llm_port      = llm_port
        self._vlm_port      = vlm_port
        self._rag           = rag
        self._graph         = graph
        self._memory        = memory

        # inQ: priority inbound queue (asyncio.PriorityQueue)
        # Items are (priority_value: int, message: OctopussyMessage)
        # CRITICAL=0 is dequeued first.
        self._inQ: asyncio.PriorityQueue[tuple[int, OctopussyMessage]] = \
            asyncio.PriorityQueue()

        # outQ: priority outbound queue (asyncio.PriorityQueue)
        # Items are (priority_value: int, message: OctopussyMessage)
        # CRITICAL=0 is dispatched first.
        self._outQ: asyncio.PriorityQueue[tuple[int, OctopussyMessage]] = \
            asyncio.PriorityQueue()

        self._loop_count: int = 0
        self._conversation_history: list[dict] = []

    @abstractmethod
    async def initialize(self) -> None:
        """CCA Lifecycle Phase 2: initialize internal state."""
        ...

    @abstractmethod
    async def inject_dependency(self) -> None:
        """
        CCA Lifecycle Phase 3: inject all required interface references.
        Called by OCLM after initialize() and before start().
        Injects: LLMPort, VLMPort, MemoryPort, RAGPort, GraphPort,
                 ToolExecutorPort, SandboxPort, TokenBudgetPort,
                 SecurityPort, MessagingContract, AgentLifecyclePort,
                 ObservabilityPort.
        """
        ...

    @abstractmethod
    async def start(self) -> None:
        """CCA Lifecycle Phase 4: start the agent's main loop."""
        ...

    @abstractmethod
    async def stop(self) -> None:
        """CCA Lifecycle Phase 5: stop the agent gracefully."""
        ...

    @abstractmethod
    async def cleanup(self) -> None:
        """CCA Lifecycle Phase 6: release all resources."""
        ...

    @abstractmethod
    async def handle_task(self, message: OctopussyMessage) -> OctopussyMessage:
        """Process a TASK message and return a RESULT message."""
        ...

    async def _call_llm(self, messages: list[dict], system: str = "") -> LLMCompletion:
        """
        Call the LLM via LLMPortAdapter (retry + circuit breaker).
        Checks token budget before call; records usage after.
        Scans prompt for injection before submission.
        """
        # Token budget check
        if not await self._token_budget.check_budget(self._agent_id, "hourly"):
            raise AgentBudgetExhaustedError(f"Agent {self._agent_id} hourly token budget exhausted")
        if not await self._token_budget.check_budget(self._agent_id, "daily"):
            raise AgentBudgetExhaustedError(f"Agent {self._agent_id} daily token budget exhausted")

        # Prompt injection scan
        full_prompt = " ".join(m.get("content", "") for m in messages)
        if not await self._security.scan_prompt(full_prompt):
            raise PromptInjectionError("Prompt injection detected; request aborted")

        request = LLMRequest(
            messages=tuple(messages),
            model=self._spec.intelligence.model,
            temperature=self._spec.intelligence.temperature,
            max_tokens=self._spec.intelligence.max_tokens,
            system=system,
            trace_id=str(uuid4()),
        )
        completion = await self._llm_port.complete(request)
        await self._token_budget.record_usage(
            self._agent_id, completion.input_tokens, completion.output_tokens
        )
        return completion

    async def _call_vlm(self, messages: list[dict], image_bytes: bytes, system: str = "") -> LLMCompletion:
        """
        Call the VLM via VLMPortAdapter (retry + circuit breaker).
        Checks token budget before call; records usage after.
        """
        if self._vlm_port is None:
            raise RuntimeError("VLM is not enabled for this agent")
        if not await self._token_budget.check_budget(self._agent_id, "hourly"):
            raise AgentBudgetExhaustedError(f"Agent {self._agent_id} hourly token budget exhausted")
        if not await self._token_budget.check_budget(self._agent_id, "daily"):
            raise AgentBudgetExhaustedError(f"Agent {self._agent_id} daily token budget exhausted")

        full_prompt = " ".join(m.get("content", "") for m in messages)
        if not await self._security.scan_prompt(full_prompt):
            raise PromptInjectionError("Prompt injection detected; VLM request aborted")

        request = LLMRequest(
            messages=tuple(messages),
            model=self._spec.intelligence.vlm_model or self._spec.intelligence.model,
            temperature=self._spec.intelligence.temperature,
            max_tokens=self._spec.intelligence.max_tokens,
            system=system,
            trace_id=str(uuid4()),
        )
        completion = await self._vlm_port.complete(request, image_bytes)
        await self._token_budget.record_usage(
            self._agent_id, completion.input_tokens, completion.output_tokens
        )
        return completion

    def _guard_loop(self) -> None:
        """Increment loop counter; raise LoopLimitExceededError if ceiling reached."""
        self._loop_count += 1
        if self._loop_count > self._spec.permissions.max_loop_iterations:
            raise LoopLimitExceededError(
                f"Agent {self._agent_id} exceeded loop limit "
                f"({self._spec.permissions.max_loop_iterations})"
            )

10. SECURITY ARCHITECTURE

10.1 Zero-Trust Enforcement Chain

Every interaction passes through the full security chain:
Inbound request (REST/gRPC/WebSocket/Channel)
│
├── mTLS verification (transport layer — RESTCommunicationPort / GRPCCommunicationPort)
├── JWT authentication (bearer token — SecurityCapability.authenticate())
├── TOTP verification (SUPERUSER/ADMIN roles — SecurityCapability)
├── RBAC authorization (deny-by-default — SecurityCapability.authorize())
│
▼
OctopussyMessage
│
├── HMAC-SHA256 signature verification (OctopussyMessage.verify_signature())
├── TTL check (OctopussyMessage.is_expired())
├── Prompt injection scan (SecurityCapability.scan_prompt())
│
▼
Agent processing
│
├── Tool permission check (allowlist — ToolRegistryCapability)
├── Sandbox execution (Bulkhead — SandboxCapability)
│
▼
Agent output
│
├── Guardrail validation (SecurityCapability.validate_output())
└── HMAC-SHA256 signature on response (OctopussyMessage.create())

10.2 HMAC Signing

Every OctopussyMessage carries a mandatory HMAC-SHA256 signature (bytes, never Optional[bytes]). Comparison uses hmac.compare_digest() (constant-time, prevents timing attacks). Key rotation is supported via SecretsCapability.rotate_secret().

10.3 Sandbox Isolation (Bulkhead Pattern)

Each agent gets its own sandbox. Tool execution is isolated per agent. A misbehaving tool in Agent A cannot affect Agent B or the host system.

10.4 User Registration and Authentication

Users must be registered with Octopussy before accessing any resource:

Registration: SecurityCapability.register_user(username, password, role) — stores bcrypt-hashed credentials in RBACPolicyStore.
Authentication: SecurityCapability.validate_user(username, password) — verifies credentials, returns JWT.
Session: UserSessionCapability.create_session(user_id, role) — creates a JWT-signed session token (RS256, TTL 8 hours).
Authorization: SecurityCapability.authorize(user_id, resource, action) — RBAC deny-by-default check on every request.

10.5 Credential Management Security

All LLM provider API tokens and credentials are stored via SecretsCapability (AES-256-GCM encrypted at rest). The flow is:

User authenticates with Octopussy (JWT).
User sets credentials via CLI (octopussy secrets set) or WebUI Credentials Manager (/ui/credentials).
Credentials are stored encrypted in the LocalEncryptedFileStore or HashiCorp Vault.
LLMRouterCapability retrieves credentials from SecretsCapability at call time.
Credentials are never logged, never stored in config files, never in environment variables.

from typing import Any, Type

class CapabilityFactory:

    Factory for creating Capability Instances.
    All Capabilities MUST be created via this factory — never instantiated directly.

14.2 OctopussyCapabilityRegistry

# octopussy/core/registry.py

from __future__ import annotations
from collections import defaultdict, deque
from typing import Any

from octopussy.core.exceptions.octopussy_exceptions import (
    CircularDependencyError,
    CapabilityNotFoundError,
)

class OctopussyCapabilityRegistry:
    """
    Authoritative catalogue of all Capability Instances.
    Detects circular dependencies at registration time (Kahn's algorithm).
    Rejects non-Capability registrations.
    """

    def __init__(self) -> None:
        self._instances:    dict[str, Any]       = {}
        self._dependencies: dict[str, list[str]] = {}

    def register(self, name: str, instance: Any, dependencies: list[str]) -> None:
        self._instances[name]    = instance
        self._dependencies[name] = dependencies

    def get(self, name: str) -> Any:
        if name not in self._instances:
            raise CapabilityNotFoundError(f"Capability {name!r} not registered")
        return self._instances[name]

    def topological_order(self) -> list[str]:
        """Kahn's algorithm — raises CircularDependencyError if cycle detected."""
        in_degree: dict[str, int] = {name: 0 for name in self._instances}
        for name, deps in self._dependencies.items():
            for dep in deps:
                in_degree[name] += 1

        queue: deque[str] = deque(
            name for name, deg in in_degree.items() if deg == 0
        )
        order: list[str] = []

        while queue:
            node = queue.popleft()
            order.append(node)
            for name, deps in self._dependencies.items():
                if node in deps:
                    in_degree[name] -= 1
                    if in_degree[name] == 0:
                        queue.append(name)

        if len(order) != len(self._instances):
            raise CircularDependencyError(
                "Circular dependency detected among registered Capabilities"
            )
        return order

ADDENDUM A — END-TO-END WORKFLOW CATALOGUE

Addendum Status: Normative

Addendum Version: 1.0 (for Octopussy 4.0.1)

This addendum specifies all primary end-to-end operational workflows of the Octopussy platform. Each workflow is described as an ordered sequence of steps, identifying the Capabilities, Ports, and message types involved at each stage. All workflows are fully consistent with CCA 0.2, the six-phase Capability lifecycle, the zero-trust security chain, and the async Actor model.

A.1 Platform Startup Workflow

Trigger: octopussy start (CLI) or asyncio.run(main()) (programmatic)
Outcome: All 26 Capability Instances are running and ready to accept work.
1.  main() invoked
    │
2.  OctopussyCapabilityRegistry instantiated
    │
3.  register_all_capabilities(registry) called
    │   └── All 26 Capabilities registered with dependency declarations
    │       Circular dependency check: Kahn's algorithm applied at registration time
    │       CircularDependencyError raised immediately if cycle detected
    │
4.  OctopussyCapabilityLifecycleManager (OCLM) instantiated with registry
    │
5.  OCLM computes topological startup order (Kahn's algorithm)
    │
6.  OCLM executes six-phase lifecycle for each Capability in topological order:
    │
    │   Phase 1 — instantiate()
    │       CapabilityFactory creates each Capability Instance exactly once
    │
    │   Phase 2 — initialize()
    │       Each Capability loads its configuration via ConfigurationPort
    │       SecretsCapability starts first (no dependencies)
    │       ObservabilityCapability starts second (Sidecar — before all others)
    │
    │   Phase 3 — inject_dependency()
    │       OCLM resolves all Requirement interfaces
    │       Injects references into each Capability via inject_dependency()
    │
    │   Phase 4 — start()
    │       Each Capability starts its async tasks and background loops
    │       MessagingCapability connects to NATS.io (or RabbitMQ)
    │       SchedulerCapability loads cron schedules
    │       ToolRegistryCapability registers all built-in tools
    │       PluginCapability loads all plugins from plugins_dir
    │
    │   [Runtime] — Capability is live and processing
    │
7.  SIGINT / SIGTERM handler registered on asyncio event loop
    │
8.  shutdown_event.wait() — platform blocks here until signal received

Invariants:

ObservabilityCapability is always the first non-root Capability to start.
SecretsCapability has no dependencies and is always first.
No Capability's start() is called before all its dependencies have completed start().
Startup order is deterministic and reproducible across restarts.


A.2 Agent Creation Workflow

Trigger: OrchestratorCapability.dispatch_task() or AgentFactoryCapability.create_agent() called directly.
Outcome: A fully initialized, running OctopussyAgent instance is registered and processing messages.
1.  Caller invokes AgentFactoryCapability.create_agent(config_path)
    │
2.  AgentFactoryCapability.AgentConfigValidator validates the Markdown/JSON config
    │   └── Raises AgentConfigValidationError if invalid
    │
3.  AgentSpecBuilder constructs a frozen AgentSpec dataclass
    │   └── AgentSpec includes: agent_id (UUID), name, agent_type,
    │       intelligence (LLM routing strategy), allowed_tools,
    │       permissions (max_loop_iterations, token_budget), memory config
    │
4.  AgentSpec cached in AgentSpecCache (keyed by agent_id)
    │
5.  AgentFactoryCapability resolves all required ports via inject_dependency():
    │   ├── LLMRouterCapability.resolve_llm_port(spec.intelligence) → LLMPort
    │   ├── MemoryCapability → MemoryPort
    │   ├── RAGCapability → RAGPort
    │   ├── GraphRAGCapability → GraphPort
    │   ├── ToolRegistryCapability → ToolExecutorPort
    │   ├── SandboxCapability → SandboxPort
    │   ├── TokenBudgetCapability → TokenBudgetPort
    │   ├── SecurityCapability → SecurityPort
    │   ├── MessagingCapability → MessagingContract
    │   └── ObservabilityCapability → ObservabilityPort
    │
6.  OctopussyAgent subclass instantiated (StatelessOneShot | StatefulLoop |
    │   ScheduledTaskDaemon) with all resolved ports injected
    │
7.  AgentLifecycleManagerCapability.transition_state(agent_id, CREATED)
    │
8.  agent.start() called:
    │   ├── asyncio.create_task(_message_loop())
    │   ├── asyncio.create_task(_outbound_dispatch_loop())
    │   └── asyncio.create_task(_heartbeat_loop())
    │
9.  AgentLifecycleManagerCapability.transition_state(agent_id, IDLE)
    │
10. AgentHandle returned to caller (contains agent_id, status endpoint)
    │
11. ObservabilityCapability logs agent creation event + emits metric
        agent_created_total{agent_type, llm_strategy}


A.3 Single-Agent Task Execution Workflow

Trigger: An OctopussyMessage of type TASK is enqueued into an agent's inQ.
Outcome: The agent produces a result and enqueues a RESULT message into outQ.
1.  Caller creates message:
    │   OctopussyMessage.create(
    │       sender_id, recipient_id, payload, priority, ttl_seconds,
    │       secret=hmac_secret
    │   )
    │   └── HMAC-SHA256 signature computed and embedded (bytes, never Optional)
    │
2.  Message enqueued into agent.inQ (asyncio.PriorityQueue)
    │   └── Priority tuple: (MessagePriority.value, message)
    │       CRITICAL=0 processed first → HIGH=1 → NORMAL=2 → LOW=3
    │
3.  _message_loop() dequeues message
    │
4.  Security chain applied (Chain of Responsibility):
    │   ├── verify_signature()     → drop + audit if HMAC invalid
    │   ├── is_expired()           → drop + audit if TTL exceeded
    │   ├── _handle_control()      → if CONTROL type: pause/resume/stop
    │   ├── scan_prompt()          → SecurityCapability.scan_prompt()
    │   │                            block + audit if injection detected
    │   └── check_budget()         → TokenBudgetCapability.check_budget()
    │                                block + audit if budget exhausted
    │
5.  AgentStateMachine.transition(IDLE → PROCESSING)
    │
6.  process_task() invoked (Template Method — subclass implements this)
    │
7.  Agent builds prompt via PromptEngineCapability (if configured)
    │
8.  Agent calls LLM:
    │   ├── TokenBudgetCapability.check_budget(agent_id, period) → abort if exhausted
    │   ├── LLMPortAdapter._call_llm(LLMRequest)
    │   │   ├── Retry + exponential backoff (up to 3 attempts)
    │   │   ├── Circuit breaker check (OPEN → fail fast)
    │   │   └── Fallback to next provider if primary fails
    │   ├── LLMCompletion returned
    │   └── TokenBudgetCapability.record_usage(input_tokens, output_tokens)
    │
9.  ReflectionEngine evaluates output (ReAct pattern)
    │   └── If reflection triggers revision: loop back to step 7
    │       LoopGuard increments counter; raises LoopLimitExceededError
    │       if max_loop_iterations exceeded
    │
10. CritiqueCapability.critique(output) called if agent has critique enabled
    │   └── If critique requests revision: loop back to step 7 (LoopGuard applies)
    │
11. SecurityCapability.validate_output(output, policy)
    │   └── Block + audit if guardrail violated
    │
12. Result OctopussyMessage.create() — RESULT type, HMAC-signed
    │
13. Result enqueued into outQ (asyncio.PriorityQueue)
    │
14. _outbound_dispatch_loop() dequeues result → MessagingCapability → recipient inQ
    │
15. AgentStateMachine.transition(PROCESSING → IDLE)
    │
16. ObservabilityCapability:
        ├── end_span(trace_id)
        ├── metric: agent_task_duration_seconds, agent_task_success_total
        └── audit: TASK_COMPLETED event


A.4 Multi-Agent Team Task Workflow — Coordinator/Worker Pattern

Trigger: TeamCapability.submit_team_task(team_id, task) called.
Outcome: Coordinator synthesizes worker results and delivers a unified team result.
1.  TeamCapability.submit_team_task(team_id, task, priority)
    │
2.  TeamOrchestrator retrieves TeamSpec from TeamStateStore
    │
3.  CoordinatorWorkerEngine activated:
    │
4.  Coordinator agent receives task via inQ (OctopussyMessage, TASK type)
    │
5.  Coordinator decomposes task into N subtasks (via LLM call, step A.3)
    │
6.  For each subtask:
    │   ├── OctopussyMessage.create(coordinator_id, worker_id, subtask, ...)
    │   └── Message enqueued into worker agent's inQ via MessagingCapability
    │
7.  Each Worker agent executes subtask (workflow A.3, steps 4–15)
    │   └── Worker enqueues RESULT message → outQ → MessagingCapability
    │           → Coordinator's inQ
    │
8.  Coordinator collects all worker RESULT messages
    │   └── Timeout enforced (team.execution.timeout_seconds)
    │       Timed-out workers: result marked as PARTIAL
    │
9.  ResultSynthesizer applies configured strategy:
    │   ├── coordinator  — Coordinator LLM synthesizes final answer
    │   ├── vote         — Majority vote across worker results
    │   └── concat       — Results concatenated in worker order
    │
10. Final RESULT OctopussyMessage created and delivered to original caller
    │
11. TeamStateStore updated; team metrics emitted:
        team_task_duration_seconds, team_task_success_total,
        team_worker_results_received_total


A.5 Multi-Agent Team Task Workflow — Pipeline Pattern

Trigger: TeamCapability.submit_team_task(team_id, task) on a Pipeline team.
Outcome: Task flows sequentially through N agents; final agent's output is the result.
1.  TeamCapability.submit_team_task(team_id, task, priority)
    │
2.  PipelineEngine retrieves ordered agent list from TeamSpec
    │
3.  Initial OctopussyMessage(task) → Agent[0].inQ
    │
4.  Agent[0] executes task (workflow A.3)
    │   └── Produces RESULT → outQ → MessagingCapability → Agent[1].inQ
    │
5.  Agent[1] receives Agent[0]'s output as its input task
    │   └── Executes (workflow A.3) → RESULT → Agent[2].inQ
    │
6.  ... repeated for Agent[2] through Agent[N-1] ...
    │
7.  Agent[N-1] produces final RESULT
    │   └── Delivered to original caller via MessagingCapability
    │
8.  PipelineEngine records pipeline completion; emits metrics
    │
    NOTE: Each inter-agent message is a fully signed OctopussyMessage.
          No direct method calls between agents at any stage.
          Each agent applies the full security chain (step A.3, step 4).


A.6 Multi-Agent Team Task Workflow — Mesh + Blackboard Pattern

Trigger: TeamCapability.submit_team_task(team_id, task) on a Mesh+Blackboard team.
Outcome: Agents collaborate peer-to-peer via a shared Redis-backed Blackboard; consensus result emerges.
1.  TeamCapability.submit_team_task(team_id, task, priority)
    │
2.  MeshBlackboardEngine broadcasts task to all mesh agents via MessagingCapability
    │   └── OctopussyMessage(TASK) → each agent's inQ simultaneously
    │
3.  Each agent reads from and writes to BlackboardStore (Redis):
    │   ├── Reads: partial results, context, peer findings
    │   └── Writes: own findings, intermediate conclusions
    │   All blackboard reads/writes are atomic (Redis transactions)
    │
4.  Agents iterate (up to max_rounds):
    │   ├── Each agent reads the current blackboard state
    │   ├── Executes its subtask (workflow A.3)
    │   └── Writes updated findings to blackboard
    │
5.  Convergence check after each round:
    │   └── If all agents agree (or max_rounds reached): proceed to step 6
    │
6.  ResultSynthesizer reads final blackboard state
    │   └── Applies configured synthesis strategy (coordinator | vote | concat)
    │
7.  Final RESULT delivered to original caller
    │
8.  BlackboardStore cleared for this task_id; metrics emitted


A.7 Tool Execution Workflow (MCP)

Trigger: Agent calls a tool inside process_task().
Outcome: Tool result returned to agent; sandbox enforced throughout.
1.  Agent invokes ToolRegistryCapability.execute_tool(tool_id, input, agent_context)
    │
2.  ToolRegistryCapability checks agent permission:
    │   └── tool_id ∈ agent_spec.permissions.allowed_tools?
    │       NO → raise ToolPermissionError (deny-by-default)
    │
3.  Input validated against tool's JSON input schema
    │   └── Raise ToolInputValidationError if invalid
    │
4.  SandboxCapability creates isolated execution environment:
    │   ├── Docker container (DOCKER profile)
    │   ├── gVisor sandbox (KUBERNETES profile)
    │   ├── seccomp filter (LOCAL profile)
    │   └── macOS sandbox (macOS LOCAL profile)
    │
5.  MCPClientAdapter constructs MCP tool call request:
    │   ├── Transport: MCP 2025-11-25 Streamable HTTP (MCPCommunicationPort)
    │   ├── Header: MCP-Protocol-Version: 2025-11-25 (mandatory)
    │   └── Endpoint: https://host:47203/mcp/tools/call
    │
6.  Tool executes inside sandbox:
    │   ├── Network access: enforced per tool's network_required flag
    │   │   (explicit allowlist; default: disabled)
    │   └── Filesystem access: restricted to sandbox working directory
    │
7.  Tool result returned via MCP Streamable HTTP response
    │
8.  Output validated against tool's JSON output schema
    │   └── Raise ToolOutputValidationError if invalid
    │
9.  SandboxCapability tears down sandbox environment
    │
10. Tool result returned to agent's process_task()
    │
11. ObservabilityCapability:
        ├── metric: tool_execution_duration_seconds{tool_id}
        ├── metric: tool_execution_success_total{tool_id}
        └── audit: TOOL_EXECUTED event (tool_id, agent_id, success, duration)


A.8 Error Handling & Dead Letter Queue (DLQ) Workflow

Trigger: Message delivery fails after dlq.max_retries attempts, or an agent raises an unhandled exception.
Outcome: Message is persisted to the DLQ; alert emitted; agent transitions to ERROR state if applicable.
SCENARIO A — Message delivery failure:

1.  MessagingCapability attempts to deliver OctopussyMessage to recipient
    │
2.  Delivery fails (recipient unreachable, queue full, network error)
    │
3.  RetryPolicy applies exponential backoff:
    │   Attempt 1 → wait 1s → Attempt 2 → wait 2s → Attempt 3 → wait 4s
    │   (max_retries = 3 per octopussy.yaml dlq config)
    │
4.  All retries exhausted:
    │
5.  DLQPort.send_to_dlq(message, reason, attempt_count) called
    │
6.  DLQStore persists message (SQLite or Redis per dlq.persist_backend)
    │   └── Stored with: original_message, failure_reason, timestamp,
    │                    sender_id, recipient_id, attempt_count
    │
7.  ObservabilityCapability:
    │   ├── metric: dlq_messages_total{reason}
    │   └── audit: MESSAGE_DLQ event
    │
8.  Alert emitted if dlq_messages_total exceeds configured threshold

SCENARIO B — Agent unhandled exception during process_task():

1.  Exception raised inside agent.process_task()
    │
2.  _message_loop() catches exception (no swallowed exceptions)
    │
3.  ObservabilityCapability.log(ERROR, exception, full trace context)
    │
4.  AgentStateMachine.transition(PROCESSING → ERROR)
    │
5.  AgentLifecycleManagerCapability.transition_state(agent_id, ERROR)
    │
6.  Original message routed to DLQ (steps 5–8 of Scenario A)
    │
7.  Recovery policy applied per AgentSpec:
    │   ├── AUTO_RESTART: AgentLifecycleManagerCapability restarts agent
    │   │   (transition: ERROR → STOPPING → STOPPED → CREATED → IDLE)
    │   └── MANUAL: agent remains in ERROR state; operator intervention required
    │
8.  ObservabilityCapability:
        ├── metric: agent_errors_total{agent_id, error_type}
        └── audit: AGENT_ERROR event

2.  PluginCapability reads updated plugin.json from plugins_dir

3.  Validation:

    │   └── plugin_type is a valid PluginType enum value
    │
4.  Old plugin instance gracefully stopped:
    │   └── If plugin has a stop() hook: called with 5s timeout

6.  New plugin instance created and initialized
    │
7.  Target Capability updated:

A.10 Scheduled Task Execution Workflow

Trigger: SchedulerCapability fires a cron trigger for a ScheduledTaskDaemon agent.
Outcome: Agent executes its scheduled task and returns to sleep until the next cron tick.
1.  SchedulerCapability.CronEngine evaluates all registered cron expressions
    │   (APScheduler backend; sub-second precision)
    │
2.  Cron expression matches current time for agent_id X
    │
3.  SchedulerCapability.TaskTrigger fires:
    │   └── OctopussyMessage.create(
    │           sender_id=SCHEDULER_SYSTEM_ID,
    │           recipient_id=agent_id,
    │           message_type=TASK,
    │           payload={"trigger": "cron", "schedule": cron_expr},
    │           priority=NORMAL,
    │           secret=hmac_secret
    │       )
    │
4.  Message enqueued into ScheduledTaskDaemon.inQ via MessagingCapability
    │
5.  Agent wakes from IDLE state:
    │   AgentStateMachine.transition(IDLE → PROCESSING)
    │
6.  Agent executes process_task() (full workflow A.3, steps 6–15)
    │
7.  Agent returns to IDLE state:
    │   AgentStateMachine.transition(PROCESSING → IDLE)
    │
8.  Agent sleeps until next cron tick (no busy-wait; asyncio event-driven)
    │
9.  ObservabilityCapability:
        ├── metric: scheduled_task_executions_total{agent_id, schedule}
        ├── metric: scheduled_task_duration_seconds{agent_id}
        └── audit: SCHEDULED_TASK_EXECUTED event


A.11 MCP Client Tool Call Workflow (Agent → External MCP Server)

Trigger: Agent calls an external MCP server tool inside process_task().
Outcome: External tool result returned to agent via MCP 2025-11-25 Streamable HTTP.
1.  Agent's process_task() invokes MCPClientAdapter.call_tool(server_url, tool_name, args)
    │
2.  MCPClientAdapter retrieves server credentials from SecretsCapability
    │
3.  HTTP session established (httpx.AsyncClient, TLS verified):
    │   └── Connection pooled per server_url (reused across calls)
    │
4.  MCP tool call request constructed:
    │   ├── Method: POST https://<server_url>/mcp/tools/call
    │   ├── Header: MCP-Protocol-Version: 2025-11-25  ← mandatory
    │   ├── Header: Authorization: Bearer <jwt_token>
    │   └── Body: {"tool": tool_name, "arguments": args}
    │
5.  Request sent via MCPCommunicationPort (Streamable HTTP transport)
    │
6.  Streaming response consumed:
    │   └── Intermediate streamed chunks buffered until complete
    │
7.  MCP-Protocol-Version response header validated:
    │   └── If version mismatch: raise MCPProtocolVersionError
    │
8.  Tool result deserialized and returned to agent
    │
9.  Retry policy applied on transient failures:
    │   └── Up to 3 attempts with exponential backoff
    │       Circuit breaker opens after 5 consecutive failures
    │
10. ObservabilityCapability:
        ├── metric: mcp_client_calls_total{server_url, tool_name, status}
        ├── metric: mcp_client_call_duration_seconds{server_url, tool_name}
        └── span: MCP_CLIENT_CALL (trace_id propagated)


A.12 Platform Shutdown Workflow

Trigger: SIGINT or SIGTERM received by the platform process.
Outcome: All Capabilities stopped gracefully in strict reverse topological order; no message loss.
1.  Signal handler sets shutdown_event
    │
2.  main() exits shutdown_event.wait()
    │
3.  lifecycle_manager.stop_all() called
    │
4.  OCLM computes reverse topological order
    │
5.  For each Capability in reverse topological order:
    │
    │   Phase 5 — stop()
    │       ├── OrchestratorCapability: stops accepting new requests
    │       ├── SchedulerCapability: cancels pending triggers
    │       ├── All AgentInstances: agent.stop() called
    │       │   ├── _stop_event.set()
    │       │   ├── _message_loop() drains inQ (processes remaining messages)
    │       │   ├── _outbound_dispatch_loop() drains outQ
    │       │   └── AgentStateMachine.transition(IDLE → STOPPING → STOPPED)
    │       ├── MessagingCapability: flushes outbound queues; disconnects from NATS/RabbitMQ
    │       └── ObservabilityCapability: flushes all pending spans/metrics/logs
    │
    │   Phase 6 — cleanup()
    │       └── Each Capability releases all resources:
    │           database connections, file handles, HTTP sessions, async tasks
    │
6.  All asyncio tasks cancelled and awaited
    │
7.  Process exits cleanly (exit code 0)
    │
    NOTE: In-flight OctopussyMessages that cannot be delivered during shutdown
          are routed to the DLQ (workflow A.8) before MessagingCapability stops.
          No message is silently dropped.


A.13 Inbound Channel Message Workflow (External User → Agent)

Trigger: A user sends a message on an external channel (Telegram, Slack, WhatsApp, etc.).
Outcome: Agent processes the message and delivers a response back to the user on the same channel.
1.  External channel delivers event to CommunicationAdapterCapability:
    │   ├── Telegram: webhook POST to CommunicationAdapterCapability REST endpoint
    │   ├── Slack: Slack Events API POST
    │   └── WhatsApp: WhatsApp Business API webhook
    │
2.  ChannelRegistry routes event to the correct adapter
    │   (TelegramAdapter | SlackAdapter | WhatsAppAdapter | ...)
    │
3.  MessageBridge converts channel-native payload → OctopussyMessage:
    │   ├── sender_id: derived from channel user identity
    │   ├── recipient_id: resolved from UserSessionCapability (session lookup)
    │   ├── payload: normalized message text / media reference
    │   └── HMAC-SHA256 signature applied
    │
4.  UserSessionCapability.validate_session(channel_user_token):
    │   └── If no active session: create_session() → JWT issued
    │
5.  SecurityCapability.authenticate() + authorize() applied
    │
6.  OctopussyMessage enqueued into target agent's inQ via MessagingCapability
    │
7.  Agent processes message (full workflow A.3)
    │
8.  Agent produces RESULT OctopussyMessage → outQ → MessagingCapability
    │
9.  CommunicationAdapterCapability.MessageBridge converts RESULT → channel-native response
    │
10. Channel adapter delivers response to user:
    │   ├── Telegram: sendMessage API call
    │   ├── Slack: chat.postMessage API call
    │   └── WhatsApp: messages API call
    │
11. ObservabilityCapability:
        ├── metric: channel_messages_received_total{channel}
        ├── metric: channel_messages_sent_total{channel}
        └── audit: CHANNEL_MESSAGE_PROCESSED event
